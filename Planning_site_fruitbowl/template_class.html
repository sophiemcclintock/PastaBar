Brief

This python program is a pasta bar ordering system. It is designed so that users will be able to easily place an online order. There will be an option to add to their order, delete from their order as well as editing it along the way. At they end they will finalise the order, choose whether or not they want delivery or pickip and enter their customer details. We are using the menu from the local Wellington pasta resturant, 1154.

Describing Relevant Implications

Usability

The usability relevant implication is making sure the user can intuitively learn how to complete the program quickly. This means ensuring the options are easy to understand and it is easy to find appropriate actions. If they want to add something to their order or delete something from their order, they are able to find a function for each of those that will guide them through it. It also means making sure the program has error prevention methods and communicates to the users about these methods. For example, if the user prints 'z' where a Y or N is required, it prints our an error message like 'Unrecognised entry'.

Functionality

The functionality relevent implication requires making sure the program works properly. This means that is does not crash, if the incorrect value is entered the program either sorts it out itself or sends back an error message. I also need to make sure it does not do incorrect arithmetic, like when it is adding up the total cost for all the pasta's added to the user's order. The last thing I want to ensure is that the variables (lists) are all reset once the order is complete. This is important because it means if the user wishes to place another order, they will not have issues with the previous order interfering with the second order.

Sustainability and Future Proofing

The sustainability and future proofing relevant implication is making sure the program is able to be updated, new features can be added easily. It also means ensuring that the progam can be handed over to another developer who can understand it quickly. This is important because if the program is passed around others need to be able to understand it otherwise it is effectively useless.

def customer_details(c):
    name = get_string("Can you please enter your name -> ", 0, 50)
    phone_number = get_string("Can you please enter your phone number -> ", 0, 15)
    print("-"*80)
    options = ["D : Delivery", "P : Pickup"]
    for x in options:
        print(x)
    user_choice = get_string("Please enter your choice -> ", 0, 1)
    if user_choice == "D":
        delivery = True
        while delivery is True:
            number = get_string("You have selected Delivery. Can you please enter your address. Number -> ", 0, 5)
            street = get_string("Street -> ", 0, 50)
            suburb = get_string("Suburb -> ", 0, 50)
            city = get_string("City -> ", 0, 50)
            postcode = get_string("Postcode -> ", 0, 4)
            address = "{} {}, {} {} {}".format(number, street, suburb, city, postcode)
            output = get_string("Your address is {}. Is this correct? (Please enter Y or N) -> ".format(address), 0, 1)
            if output == "Y":
                delivery = False
            elif output == "N":
                delivery = True
        output = "Address: {}, Phone Number: {}, Name: {}".format(address, phone_number, name)
        c.extend([name, phone_number, address])
        print(output)
    elif user_choice == "P":
        print("You have selected Pickup")
        output = "Phone Number: {}, Name: {}".format(phone_number, name)
        c.extend([name, phone_number])
        print(output)

def finalising_order(o, c):
    review_order(o)
    confirm = get_string("Is this order correct? Please enter Yes (Y) or No (No) -> ", 0, 1)
    correcting_order = True
    while correcting_order is True:
        if confirm == "Y":
            customer_details(c)
            print("Your order has been finalised")
            o.clear()
            c.clear()
            order_or_quit = order_again()
            return order_or_quit
            correcting_order = False
        elif confirm == "N":
            correcting_order = False
            return True
        else:
            print("Unrecognised entry")
    
def order_again():
    output = get_string("Would you like to place another order? Please enter Yes (Y) or No (No) -> ", 0, 1)
    if output == "Y":
        return True
    elif output == "N":
        print("Thank you for ordering your meal. The program will terminate")
        return False
    else:
        print("Unrecognised entry")

Addressing Relevant Implications

Usability

Addressing the usability relevant implication means creating validation functions to ensure appropriate entry. This is important because if the user entered an incorrect entry, and I didn't have validation, the program would crash. I want to be able to tell the user what they did wrong and give them another chance to type in the needed input. Another way of ensuring the usability is using a simple menu system, like a home screen for the user. This ensures that while the user can branch out to different functions and complete tasks like adding something to their menu or deleting something, they are also able to return to the menu.

Functionality

Addressing the functionality relevent implication means carrying out systematic testing throughout. This is so important because if there is an error or something is not working, it means we can fix it as soon as possible instead of it potenially becoming a much bigger problem. It is necessary to do a test at the end of each sprint then run a full user test with a real user to test the program under unpredictable conditions. Throughout my planning you can see evidence of tests at the end of each sprint and I have also attached a video running through the code with Mr Khouri. This is so benefical to whole coding experience as it means the coding is less likely to have errors.

Sustainability and Future Proofing

To address the sustainability and future proofing relevant implication I have made sure to use a function based build. This means that each feature is a function and this makes the program so much easier to follow because instead of one big chunk of code, it is split into lots of different functions. For example, in the finalising function, there are a few different functions inside that paticular function, such as collecting the customer details and asking if the customer would like to order again. To ensure the program is future proof I have also done lots of DocTyping and commenting so if the user doesn't understand a certain part of the code, they can just double check the comments.
